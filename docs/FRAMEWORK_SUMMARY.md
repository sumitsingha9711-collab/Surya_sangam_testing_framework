# Surya Sangam Automation Framework

## Purpose

This project is a Python, Selenium, and Pytest Page Object Model (POM) framework for the Surya Sangam website. It covers the homepage, About page, shared navigation, CTA buttons, responsive behavior, footer and social links, and the Rooftop Solar Estimator.

## Structure

```text
surya_sangam_testing/
|-- pages/       Page objects and reusable page components
|-- locators/    Centralized Selenium locators
|-- tests/       Independent Pytest scenarios
|-- utils/       Driver, waits, screenshots, reports, and email support
|-- docs/        Framework and learning documentation
|-- conftest.py  Browser fixture, failure screenshots, and report hooks
|-- pytest.ini   Test discovery and markers
|-- README.md    Quick-start instructions
```

There is one canonical page object per primary feature:

- `HomePage` for the homepage.
- `AboutPage` for the About page.
- `CalculatorPage` for the Solar Calculator section.
- `Header` for shared navigation.
- Reusable components for footer, contact form, and responsive checks.

Obsolete parallel page objects and broken test-only reporters were removed. The active report implementation is `utils/report_generator.py`.

## Calculator Workflow

The live calculator is a homepage section at:

```text
https://www.suryasangam.com/#surya-calculator
```

The current required controls are:

- Address autocomplete input.
- Average Units numeric input.
- Average Bill numeric input.
- `Get Advanced Quote` button.

`CalculatorPage.enter_location()` types the address and selects a visible autocomplete suggestion when the site provides one. This is required for valid submissions because the site tracks a selected address, not only the input text.

The page object also supports optional dropdowns, radio buttons, checkboxes, reset controls, inline messages, alerts, and estimator result cards so future calculator fields can be added without moving selectors into tests.

## Test Coverage

`tests/test_calculator.py` covers:

- Page URL, title, heading, and required controls.
- Empty submission validation.
- Alphabetic, symbolic, negative, oversized, decimal, whitespace, and malformed values.
- Multiple valid datasets and estimator result visibility.
- Reset or fallback clear behavior.
- Calculator button visibility and enabled state.

Invalid input is considered handled when the application either reports validation or remains safe without producing a result. This reflects the current live behavior for some free-form address values.

Other test modules cover homepage, About, navigation, CTA buttons, footer, social links, image loading, and responsive layout.

## Shared Infrastructure

- `BasePage` provides common click, typing, clearing, text, scrolling, and explicit-wait operations.
- `utils/wait_utils.py` provides visibility, clickability, presence, and document-ready waits.
- `utils/driver_factory.py` creates Chrome through webdriver-manager.
- `conftest.py` creates one driver per test, captures failure screenshots, and records results.
- `utils/screenshot.py` writes failure screenshots under `screenshots/`.
- `utils/report_generator.py` writes `reports/execution_report.txt` with status, duration, category, failure details, and screenshot evidence.

The framework does not use `time.sleep()`.

## Running Tests

```powershell
pip install -r requirements.txt
pytest
pytest -m calculator
pytest -m homepage
pytest -m about
```

Reports are generated after every run. Failure screenshots are generated by the existing Pytest hook and do not require test-specific reporting code.
## Contact Page Coverage

Phase 5 adds Contact page automation through `ContactPage` and `ContactLocators` without changing the POM architecture.

The current live Contact form contains required name, email, phone, average electricity bill, and address fields, plus an optional message field. The page object uses native browser validation, inline or alert messages, and the existing explicit-wait helpers. It also exposes contact email, phone, address, CTA, reset, success, and optional Google Map checks.

Run the module with:

```powershell
pytest -m contact
```

Failure screenshots and report entries continue to be handled centrally by `conftest.py`; no Contact-specific reporting system was added.
